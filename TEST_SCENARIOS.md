# User Test Scenarios

Этот файл описывает ручные пользовательские сценарии для проверки MVP на уровне продукта, а не только отдельных API.

## Подготовка

1. Поднять backend и Postgres:

```bash
docker compose -f infra/docker-compose.yml --env-file infra/.env up -d --build
```

2. Проверить health:

```bash
curl http://localhost:8080/api/health
```

3. Убедиться, что доступны demo invite code:

- `PARENT-DEMO`
- `CHILD-DEMO`

4. Для локального mixed-режима можно запускать backend отдельно:

```bash
docker compose -f infra/docker-compose.local.yml --env-file infra/.env up -d
PORT=8081 ./gradlew :backend:run
```

## Сценарий 1. Первый вход родителя

Цель:
- проверить onboarding
- проверить `register-device`
- проверить сохранение сессии

Шаги:
1. Открыть web client или desktop client.
2. Ввести `Server Base URL`.
3. Выбрать `Register`.
4. Ввести invite code `PARENT-DEMO`.
5. Ввести имя устройства.
6. Нажать `Register Device`.

Ожидание:
- вход успешен
- открывается экран contacts
- текущий пользователь отображается как `Parent`
- в БД появляется `users/devices/auth_tokens` запись

## Сценарий 2. Первый вход ребёнка

Цель:
- проверить второй пользовательский профиль
- проверить family-scoped contacts

Шаги:
1. Открыть второй клиент, лучше на другой платформе.
2. Выбрать `Register`.
3. Ввести invite code `CHILD-DEMO`.
4. Ввести имя устройства.
5. Нажать `Register Device`.

Ожидание:
- вход успешен
- у ребёнка в contacts виден `Parent`
- у родителя в contacts виден `Child`

## Сценарий 3. Повторный вход через Login

Цель:
- проверить `login`
- проверить reuse уже зарегистрированного устройства

Шаги:
1. Разлогиниться на уже зарегистрированном устройстве.
2. Выбрать `Login`.
3. Ввести тот же invite code и то же имя устройства.
4. Нажать `Login`.

Ожидание:
- вход успешен
- создаётся новый auth token
- профиль и family остаются прежними

## Сценарий 4. Отправка текстового сообщения

Цель:
- проверить `messages/send`
- проверить появление сообщения в чате

Шаги:
1. У родителя открыть чат с ребёнком.
2. Ввести короткий текст.
3. Нажать `Send text`.

Ожидание:
- сообщение появляется локально сразу
- после sync остаётся в истории
- у получателя сообщение появляется после polling/sync

## Сценарий 5. Quick actions

Цель:
- проверить быстрые действия

Шаги:
1. Открыть чат.
2. Нажать `AT_SCHOOL`, `PICK_ME_UP` или другой quick action.

Ожидание:
- quick action отправляется как отдельное сообщение
- у получателя оно отображается в истории
- в БД `messages.type = QUICK_ACTION`

## Сценарий 6. Incremental sync

Цель:
- проверить `sync?since_id=...`

Шаги:
1. Отправить несколько сообщений с одного клиента.
2. Второй клиент оставить открытым.
3. Наблюдать, как сообщения приходят без полного перезапуска клиента.

Ожидание:
- новые сообщения приходят через polling
- `syncCursor` двигается вперёд
- уже полученные сообщения не дублируются

## Сценарий 7. Receipts: delivered

Цель:
- проверить `mark-delivered`

Шаги:
1. Отправить сообщение от родителя ребёнку.
2. Открыть чат у ребёнка.

Ожидание:
- у отправителя статус сообщения становится не ниже `DELIVERED`
- в БД у receipt получателя появляется `delivered_at`

## Сценарий 8. Receipts: read

Цель:
- проверить `mark-read`

Шаги:
1. Получить сообщение на втором клиенте.
2. Открыть соответствующий чат.
3. Явно вызвать `Mark read` через UI, если нужно.

Ожидание:
- у отправителя статус сообщения становится `READ`
- в БД у receipt получателя появляется `read_at`

## Сценарий 9. Сохранение сессии после перезапуска клиента

Цель:
- проверить session restore

Шаги:
1. Войти в клиент.
2. Закрыть приложение.
3. Открыть приложение снова.

Ожидание:
- клиент не просит логин заново
- профиль восстанавливается
- contacts/history подхватываются из локального состояния и sync

## Сценарий 10. Offline-first минимальное поведение

Цель:
- проверить local queue и восстановление после сетевого сбоя

Шаги:
1. Открыть чат.
2. Временно отключить backend или сеть.
3. Отправить текст.
4. Вернуть backend/сеть.

Ожидание:
- сообщение сначала локально видно как pending
- после восстановления сети сообщение уходит на сервер
- дубликаты не появляются

## Сценарий 11. Location flow

Цель:
- проверить location message и location event

Шаги:
1. Открыть чат.
2. Нажать `Share location`.

Ожидание:
- если платформа не даёт координату, клиент показывает понятный отказ
- если координата доступна, появляется location message в чате
- backend принимает `/api/location/share`
- в sync приходят location-related данные

## Сценарий 12. Web client smoke

Цель:
- проверить практическую пригодность Web JS

Шаги:
1. Запустить:

```bash
./gradlew :client:composeApp:jsBrowserDevelopmentRun
```

2. Открыть `http://localhost:8082`.
3. Пройти вход и отправить сообщение.

Ожидание:
- onboarding работает
- логин/регистрация проходят
- чат работает
- CORS не ломает auth и messaging flow

## Сценарий 13. Desktop client smoke

Цель:
- проверить launchability desktop

Шаги:
1. Запустить:

```bash
./gradlew :client:composeApp:run
```

2. Пройти login/register.
3. Отправить сообщение.

Ожидание:
- окно открывается
- клиент подключается к backend
- chat flow работает

## Сценарий 14. Android smoke

Цель:
- проверить launchability Android после ручного запуска

Шаги:
1. Установить приложение на эмулятор/устройство.
2. Указать base URL:
   - для эмулятора Android обычно `http://10.0.2.2:8081`
3. Пройти onboarding.

Ожидание:
- приложение стартует
- auth работает
- контакты загружаются
- сообщения отправляются

## Сценарий 15. Invite limits

Цель:
- проверить ограничения invite code

Шаги:
1. Использовать одноразовый invite до исчерпания.
2. Повторить регистрацию ещё раз.

Ожидание:
- backend отклоняет регистрацию
- пользователь получает понятную ошибку

## Сценарий 16. Family isolation

Цель:
- проверить, что пользователь видит только свою family

Шаги:
1. Создать отдельную семью или отдельный набор invite code.
2. Войти пользователями из разных семей.
3. Проверить контакты и чаты.

Ожидание:
- чужие пользователи не видны
- отправка сообщений между семьями невозможна

## Сценарий 17. Swagger/API smoke

Цель:
- проверить ручную диагностику backend

Шаги:
1. Открыть Swagger:

```text
http://localhost:8081/api-docs/swagger
```

2. Получить token через `register-device` или `login`.
3. Нажать `Authorize`.
4. Проверить `/api/profile/me`, `/api/contacts`, `/api/messages/sync`.

Ожидание:
- token подставляется
- protected endpoints работают из Swagger

## Сценарий 18. Первый VPS smoke

Цель:
- проверить минимальную deploy-пригодность

Шаги:
1. Поднять compose на VPS.
2. Проверить внешний доступ к backend.
3. Открыть web/desktop client и направить его на VPS base URL.
4. Пройти auth и send/sync flow.

Ожидание:
- backend доступен извне
- клиенты реально работают не только локально

## Минимальный smoke checklist перед демонстрацией

- backend health отвечает
- родитель может войти
- ребёнок может войти
- contacts загружаются
- text message доходит
- quick action доходит
- sync работает
- receipts обновляются
- web client работает
- desktop client работает
